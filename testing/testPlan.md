## 1\. Введение <a name = "1"></a>
В данном документе представлен план тестирования приложения CodeBuilder. Цель тестирования - проверка соответствия между реальным поведением программы и её ожидаемым поведением на основе требований.


## 2\. Объект тестирования <a name = "2"></a>
Объект тестирования - плагин "CodeBuilder". Данное приложение разработано с целью упростить разработку программного обеспечения. 

Атрибуты качества по ISO 25010:  

1. Функциональная пригодность  
 - Функциональная полнота  
 - Функциональная корректность  
 - Функциональная целесообразность.<br>
2. Удобство использования
 - Эстетика пользовательского интерфейса  
 - Работоспособность
 - Защита от ошибок пользователя
 - Доступность

 Приложение должно выполнять основные функции, обозначенные в SRS. Также приложение должно соответствовать нефункциональным требованиям, обозначенным в SRS.

## 3\. Риски <a name = "3"></a>
  - Обновление версии Java;
  - Обновление среды Eclipse.


## 4\. Аспекты тестирования <a name = "4"></a>

#### Тестируемые функции, заявленные в SRS:

1.  Просмотр существующей структуры класса
2.  Декларирование методов
3.  Декларирование переменных класса
4.  Удаление элемента класса
5.  Декларирование переменных метода
6.  Удаление переменных метода
7.  Добавление функционального блока в структуру метода
8.  Параметризация функционального блока в структуре метода
9.  Генерация кода на основе внесенных изменений
10.  Сохранение файла

#### Тестируемые нефункциональные требования:
1.  Приложение должно быть доступно для последней версии платформы Eclipse 
2.  Пользователь имеет возможность изменять размер размер элементы интерфейса.

## 5\. Подходы к тестированию <a name = "5"></a>
Для тестирования пользовательского интерфейса приложения используется ручное тестирование. Для тестирования алгоритмов генерации кода и парсинга XML файлов используется unit-тестирование с помощью фреймворка  [JUnit](https://junit.org/junit5/).
